# MiniDiary

MiniDiaryは、日々の出来事や感情を記録し、他のユーザーと共有できるSNS風の日記アプリです。  
MERNスタック（MongoDB / Express / React / Node.js）を用いて、フロントエンド、バックエンド、データベース連携、デプロイまで学習を兼ねて制作しました。

## デモ

- デモURL: <https://minidiary.onrender.com>
- サンプルユーザー名: `サンプル`
- サンプルパスワード: `abcd1234`

> Renderの無料プランでデプロイしているため、初回アクセス時はサーバー起動に少し時間がかかる場合があります。

## アプリ概要

MiniDiaryでは、ユーザーが日記を投稿し、画像を添付したり、他のユーザーの日記にいいね・コメントをしたりできます。  
フォローしたユーザーの投稿を表示するタイムラインや、いいね・コメント・フォロー時の通知機能も実装しました。

個人の日記投稿を中心に、ユーザー同士の簡単な交流もできるアプリを目指して制作しました。

## 主な機能

| 機能 | 内容 |
|---|---|
| ユーザー認証 | 新規登録、ログイン、ログアウト、セッションによるログイン状態の保持 |
| 日記投稿 | テキスト投稿、画像付き投稿、投稿削除 |
| タイムライン | 全体タイムライン、フォロー中ユーザーのタイムライン |
| いいね | 投稿へのいいね、いいね解除、いいねした投稿の表示 |
| コメント | 投稿へのコメント、コメント削除 |
| フォロー | ユーザーのフォロー、フォロー解除 |
| 通知 | いいね、コメント、フォロー時の通知表示 |
| 検索 | ユーザー名検索、おすすめユーザー表示 |
| プロフィール | プロフィール画像、ユーザー名、自己紹介文の編集 |
| レスポンシブ対応 | PCではサイドバー、スマートフォンではボトムナビゲーションを表示 |

## 技術スタック

| 領域 | 使用技術 |
|---|---|
| フロントエンド | React / React Router / Redux Toolkit / Axios |
| バックエンド | Node.js / Express |
| データベース | MongoDB / Mongoose |
| 認証 | express-session / connect-mongo / bcrypt |
| 画像管理 | Cloudinary / multer |
| UI | CSS / Material UI Icons / styled-components |
| デプロイ | Render |

```text
  |
  | Axios
  v

  |
  | Mongoose
  v
MongoDB

画像アップロード: Frontend -> Express -> Cloudinary
セッション管理: express-session -> MongoDB Store
```

## 工夫した点

- セッション情報をMongoDBに保存し、ページを再読み込みしてもログイン状態を復元できるようにしました。
- 投稿・コメント・フォロー・通知など、SNSでよく使われる機能を実装しました。
- 投稿やコメントの取得時にユーザー情報も付与し、タイムライン上で投稿者やコメント投稿者が分かるようにしました。
- PCとスマートフォンでナビゲーションを切り替え、画面幅に応じて使いやすいUIになるよう調整しました。
- 投稿、コメント、通知、画像削除などの操作では、ログインユーザー本人の権限をサーバー側で確認するようにしました。

## セキュリティ面で意識したこと

- パスワードはbcryptでハッシュ化して保存しています。
- ログイン後はセッションを再生成し、セッション固定攻撃のリスクを下げています。
- 投稿作成やコメント作成では、クライアントから送られたユーザーIDではなく、サーバー側のセッション情報からユーザーを判定しています。
- 投稿・コメント・プロフィール・通知・画像削除では、操作対象がログインユーザー本人のものかを確認しています。
- 本番環境ではCookieに`httpOnly`、`secure`、`sameSite`を設定しています。

## 苦労した点

- 投稿、コメント、ユーザー情報を別々に管理しているため、タイムライン表示時に必要なユーザー情報をどう結合するかに苦労しました。
- 画像投稿では、Cloudinaryへのアップロード結果を投稿データと紐づけ、削除時にも画像を消せるようにする必要がありました。
- ログイン状態の復元、未ログイン時のリダイレクト、エラー時の画面遷移など、フロントエンドとバックエンドの状態を揃える部分に苦労しました。
- 当初はクライアントから送られるユーザーIDに依存した処理がありましたが、サーバー側のセッションを基準にした認可処理へ見直しました。
